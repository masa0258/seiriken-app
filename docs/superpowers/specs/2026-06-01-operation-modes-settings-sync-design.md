# 運用向け 役割別モード＋設定クラウド同期 設計

作成日: 2026-06-01

## 目的

整理券アプリ（`index.html`）を実運用に載せるため、1台の端末に全機能が露出している現状を改め、端末ごとに役割を固定する。顧客タブレットは「受付」だけ、モニターは「呼び出し表示」だけ、スタッフ端末は「呼び出し管理＋記録・集計＋（隠れた）設定」を扱う。あわせて店舗内容設定をクラウド経由で同期し、どの端末で変更しても他端末へ伝播するようにする。設定はITリテラシーの低いスタッフが誤って変更しないよう、二重の壁（長押し＋PIN）の奥に隠す。

## コンセプト

**1ファイル・URLで役割を固定。設定は奥に隠してクラウドで一元管理。** ファイルは `index.html` 1枚のまま、`?mode=` で表示画面を絞る。設定は店舗内容のみクラウド同期し、接続秘密キーは各端末ローカルに留める。

## スコープ

### 今回やること
- `index.html` のナビ（現在の5タブ）を廃止し、URLパラメータ `?mode=` による役割別表示へ変更。
- 役割モード：`reception`（受付のみ）／`display`（呼び出し表示のみ）／`staff`（呼び出し管理＋記録・集計）。パラメータなし・不明値は安全側で `reception` にフォールバック。
- スタッフ画面から「設定」タブを撤去し、タイトル長押し（2秒）＋PIN（4桁）の隠し動線で設定画面を開く方式に変更。
- 店舗内容設定（店名／ヘッダー／フッター／QRリンク／待ち時間表示の有無）とPINを、Supabase経由でクラウド同期。読み取りはanonキー、書き込みは秘密キー必須の新RPC `publish_config`。
- 起動時にクラウドから設定を取得して適用、失敗時はlocalStorageキャッシュへフォールバック。保存時はローカル保存＋（秘密キーがあれば）クラウド書き込み。
- index.html に既定のSupabase URL＋anonキー（公開安全）を埋め込み、受付・モニターを設定レスで動作させる。

### 今回やらないこと（YAGNI）
- ログイン／ユーザー管理／役割ごとのアカウント認証。
- PIN総当たり対策（数回失敗でロック等）。今回はうっかり変更防止のUI壁のみ。
- 設定のリアルタイム双方向同期や競合解決。最後の書き込み勝ち（last-write-wins）で割り切る。
- status.html（顧客QRサイト）・queue.js・printer.js・status.js のロジック変更。
- 接続情報（URL／anon／秘密キー）自体のクラウド同期（鶏卵問題のため対象外）。
- 役割モードの追加（キッチン用等）、多言語切替、テーマ切替。

## 確定した方針

- **役割固定**: URLパラメータ `?mode=`。各端末は対応URLをホーム画面に登録。
- **設定の保護**: スタッフ画面に設定ボタンを出さない。タイトル2秒長押し → PIN入力 → 設定画面。初期PIN `0000`。
- **同期範囲**: 店舗内容設定＋PINのみクラウド同期。接続情報・プリンターは端末ローカル。
- **接続のブートストラップ**: index.html に既定URL＋anonを埋め込み。秘密キーはスタッフ設定でのみ入力。
- **変更範囲**: 主に `index.html`（モード分岐・隠し設定・同期）＋ Supabaseのセットアップ用SQL（`publish_config` RPC・読み取りRLS）。

## 役割モード仕様

`index.html` を読み込み、URLの `mode` クエリで表示を決定する。

| `?mode=` | 用途 | 表示 |
|----------|------|------|
| `reception` | 顧客タブレット | 受付画面のみ。ナビ非表示・全画面。 |
| `display` | モニター | 呼び出し表示画面のみ。ナビ非表示・全画面。 |
| `staff` | スタッフ端末 | 「呼び出し管理」「記録・集計」の2タブ。設定ボタンなし。 |
| なし／不明 | フォールバック | `reception` 扱い（最小権限・情報露出ゼロ）。 |

- 既存の5タブ用ナビ（`<nav>`）は撤去。スタッフモードのみ管理／記録集計の2タブを表示。
- 受付・モニターには秘密キー・スタッフ機能・設定動線が一切露出しない。
- 各端末に登録するURL（本番）:
  - 受付: `https://masa0258.github.io/seiriken-app/index.html?mode=reception`
  - 表示: `https://masa0258.github.io/seiriken-app/index.html?mode=display`
  - スタッフ: `https://masa0258.github.io/seiriken-app/index.html?mode=staff`

## 隠し設定＋PIN仕様

- スタッフ画面のタイトル（例:「呼び出し管理」見出し）を **2秒長押し**（`touchstart`/`mousedown` 開始 → タイマー、`touchend`/`mouseup`/`touchmove` で解除）すると、PIN入力ダイアログを表示。通常タップ・クリックでは無反応。
- PINは4桁。正解で設定画面表示、不正解で閉じる。失敗ロックは設けない（YAGNI）。
- 初期PINは `0000`。設定画面で変更可能。PINはクラウド同期対象（1台で変更すれば他端末へ伝播）。
- 設定画面の構成（既存の設定セクションを流用・再配置）:
  - 店舗内容設定: 店名／ヘッダー／フッター／QRリンク／待ち時間表示の有無 …… クラウド同期
  - PIN変更 …… クラウド同期
  - オンライン共有 接続: Supabase URL／anonキー／書き込み秘密キー …… 端末ローカル（秘密キーはここだけ）
  - プリンター: USB/Bluetooth接続・テスト印刷・プレビュー …… ローカル
  - 「閉じる／戻る」でスタッフ通常画面へ復帰。

## 設定クラウド同期仕様

### 同期対象
- 同期: `storeName` / `headerMessage` / `footerMessage` / `qrUrl` / `showWaitEstimate` / `pin`
- 非同期（ローカル）: Supabase接続情報（URL/anon/secret）、プリンター接続状態

### データの持ち方
- Supabase に設定用の固定1行（例: ROW_ID `'main'`）を持つテーブルを用意。店名等は秘密でないため、anonキーで該当行を**読み取り可**（RLS）。
- 書き込みは秘密キー必須の新RPC `publish_config`（`publish_status` と同じ SECURITY DEFINER ＋ シークレットチェックの作法）。
- PINをanon可読の行に置く保護の弱さは、「うっかり変更防止のUI壁」という位置づけで割り切る（外部攻撃者対策ではない）。

### 動作フロー
1. **読み込み（全モード共通）**: 起動時、接続情報があればクラウドから設定取得 → 適用。失敗時は localStorage キャッシュへフォールバック。成功時はキャッシュ更新。
2. **保存（スタッフ設定のみ）**: 「設定を保存」で (a) localStorage 保存、(b) 秘密キーがあれば `publish_config` でクラウド書き込み。
3. **伝播**: 他端末は次回起動時、または既存の5秒ポーリングに相乗りして最新設定を取得・適用（例: 受付の印刷店名、表示画面の店名が自動更新）。last-write-wins。

### 接続のブートストラップ
- index.html に既定の Supabase URL＋anonキー（公開安全。status.html と同様に埋め込み済みの実績あり）を持たせ、受付・モニターは設定なしで読み取り可能にする。秘密キーはスタッフ設定でのみ入力。

## 維持する技術的制約（重要）

- **DOM操作は `textContent`/`createElement`/`appendChild`/`createTextNode` のみ**。`innerHTML` 不使用（セキュリティフックでブロック）。
- 外部ライブラリ・フォント・CSSを新たに読み込まない。自己完結を維持。
- queue.js / printer.js / status.js の純粋関数ロジックは不変。既存テスト（queue/printer/status の各テスト）がPASSし続けること。
- 秘密キー（書き込み用）は受付・モニター端末に絶対に露出させない。スタッフ設定（PINの奥）にのみ存在。
- Supabase publishable/anon キー・URL は公開安全。埋め込み可。

## データフロー

- 受付/管理/表示の整理券ロジックは現状維持（localStorage `seiriken-state`、必要に応じ `publish_status` で queue_status をクラウド配信）。
- 追加されるのは設定の同期経路のみ: スタッフ設定保存 → `publish_config`（秘密キー）→ Supabase 設定行 → 各端末が anon で読み取り → 適用。

## エラー・端の処理

- `?mode=` 不明/なし → 受付にフォールバック。
- クラウド設定取得失敗 → localStorage キャッシュを使用、画面は崩さない。
- 秘密キー未設定でスタッフが保存 → ローカル保存のみ（クラウド書き込みはスキップ、その旨をステータス表示）。
- PIN不正 → 設定を開かず閉じる。

## テスト

- **自動**: 既存の queue/printer/status のテストが全てPASS（ロジック無変更の担保）。モード判定など新規に純粋関数化できる部分（例: `resolveMode(search)`）はNode vm ハーネスでテスト追加。
- **静的**: `grep -c innerHTML index.html` が 0。
- **手動**: 3モードのURLで各端末表示、長押し＋PINでの設定開閉、設定保存→別端末での反映、秘密キー無し端末での非露出を確認。

## 手動検証

1. `?mode=reception` / `?mode=display` / `?mode=staff` の各URLで、想定どおりの画面だけ表示される。パラメータなしが受付になる。
2. スタッフ端末でタイトル長押し→PIN `0000` で設定が開く。通常タップでは開かない。
3. 設定で店名を変更・保存 → 受付/表示/別スタッフ端末で店名が更新される。
4. 受付・モニター端末に秘密キーが露出していない（設定動線が無い）。
5. オフライン/取得失敗時にキャッシュで表示が維持される。
6. queue/printer/status テスト全PASS、`innerHTML` 0。
